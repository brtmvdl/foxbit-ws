Authentication

Before being able to sign any requests, you must create an API key via the Foxbit platform. After creating a key, you need to write down 2 pieces of information, Key and Secret. The Key and Secret are generated and provided by Foxbit. Please note that these two pieces of information cannot be recovered once lost. If you lost this information, please create a new API key.

Creating a Request

All private REST requests must contain the following headers:

X-FB-ACCESS-KEY The API key as a string.
X-FB-ACCESS-TIMESTAMP A timestamp for your request, generated with Date.now().
X-FB-ACCESS-SIGNATURE The hex-encoded signature. See details below.

Generating the hex-encoded signature

Use API Secret to encrypt the prehash string with sha256 HMAC. The request body is a JSON string and needs to match the parameters passed to the API.

Example using JavaScript

GET Example

const apiSecret = 'YOUR API SECRET GOES HERE';
const payload = {
  method: 'GET',
  url: '/rest/v3/trades',
  query: 'market_symbol=btcbrl'
};
const timestamp = Date.now();
const prehash = `${timestamp}${payload.method}${payload.url}${payload.query}`;
const signature = CryptoJS.HmacSHA256(prehash, apiSecret).toString();

console.log('signature:', signature);

POST Example

const apiSecret = 'YOUR API SECRET GOES HERE';
const payload = {
  method: 'POST',
  url: '/rest/v3/orders',
  body: {
    // ... body data
  }
};
const timestamp = Date.now();
const rawBody = JSON.stringify(payload.body);
const prehash = `${timestamp}${payload.method}${payload.url}${rawBody}`;
const signature = CryptoJS.HmacSHA256(prehash, apiSecret).toString();

console.log('signature:', signature);

Rate Limit

Our rate limit allow a maximum requests (limit) in a time window (ttl).
For public endpoints, we use your IP address as a key. But for private endpoint's, your SN is used as an indentifier.

If an endpoint does not have rate limit information in its documentation, then it will be using our global rate limit values:

Limit Ttl Block duration
300 req 10 sec. 10 sec.

Endpoints with a specific rate limit (described in their documentation) will override this values.

Idempotent Requests

The API supports idempotency for safely retrying requests without accidentally performing the same operation twice. This is useful when an API call is disrupted in transit and you do not receive a response. For example, if a request to create an order does not respond due to a network connection error, you can retry the request with the same idempotency key to guarantee that no more than one order is created.

To perform an idempotent request, provide an additional X-Idempotent: <key> header to the request.

Foxbit's idempotency works by saving the resulting status code and body of the first request made for any given idempotency key, regardless of whether it succeeded or failed. Subsequent requests with the same key return the same result, including 500 errors.

An idempotency key is a unique value generated by the client which the server uses to recognize subsequent retries of the same request. How you create unique keys is up to you, but we suggest using V4 UUIDs, or another random string with enough entropy to avoid collisions. Idempotency keys can be up to 36 characters long.

Keys are eligible to be removed from the system automatically after they're at least 72 hours old, and a new request is generated if a key is reused after the original has been pruned.

Results are only saved if an API endpoint started executing.

In case of concurrent request, only one of them will be process, the other request will return status code 409 and an error, the second request does not get saved by the idempotency.

All POST requests accept idempotency keys. Sending idempotency keys in GET, PUT or DELETE requests has no effect and should be avoided, as these requests are idempotent by definition.

Tutorials

How to manage a local copy of the order book

Follow the steps below in sequential order

  1. Open a websocket stream to receive order book updates. When the connection opened, the returned data must be buffered. See documentation. The first numeric item (so called ID) in each update refers to an incremental number that represents the current state of the order book.
  2. Get a snapshot of the order book from our REST API. See documentation. The property sequence_id returned refers to an incremental number that represents the current state of the order book.
  3. Update your local copy of the order book each time the ID of an order book update is strictly equals to sequence_id + 1.

Important

  • While listening to the stream, events can only be processed if their ID remains sequential. If you find a gap between IDs, you need to restart the process from the beginning.
  • The data at each event contains the most recent quantity for a price level. If this quantity is zero, please, remove the price level.

Required API Key permissions: trade
Rate Limit: This endpoint has the limit of 30 requests per 2 seconds. An order, after created can be presented in the following states:

State Description
ACTIVE The order is in the matching queue, waiting for deal
PARTIALLY_FILLED A part of the order has been filled and it is in the matching queue, waiting for deal
FILLED The order is already traded and not in the matching queue anymore
PARTIALLY_CANCELED The order is not in the matching queue anymore The order was partially traded, but the remaining was canceled
CANCELED The order is not in the matching queue anymore and is completely canceled There is no trade associated with this order
